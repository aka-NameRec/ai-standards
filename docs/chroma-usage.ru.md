# Руководство по использованию Chroma

Англоязычная оригинальная версия: [chroma-usage.md](chroma-usage.md)

Это руководство объясняет, как использовать фичу `chroma` из `ai-standards` в downstream-проектах.

`chroma` стандартизует семантический поиск по исходному коду репозитория как слой, намеренно отдельный от оперативной памяти ConPort и retrieval-слоя документации Basic Memory.

## Цели

Используйте `chroma`, когда хотите, чтобы агент или команда:

- выполняли семантический поиск по коду в большом или мульти-репозиторном пространстве
- держали retrieval по коду изолированным от retrieval по документации и от оперативного контекста
- избегали устаревших результатов, обновляя индекс перед запросом
- возобновляли прерванные сборки индекса без пере-embedding неизменённых файлов

Типичные результаты:

- более быстрая навигация по незнакомому коду
- более узкие цели исследования перед exact search или build-верификацией
- меньше путаницы между «тем, что говорят доки» и «тем, что делает код»

## Что покрывает фича

Фича стандартизует общую политику для:

- использования Chroma как слоя семантического поиска по исходным файлам
- удержания индекса кода отдельным от хранилищ ConPort и Basic Memory
- freshness-gated-запросов (обновление перед запросом, блокировка при ошибке обновления)
- инкрементальной индексации по content-hash с atomic resumable-манифестом
- локального `PersistentClient` как режима развёртывания по умолчанию

Она намеренно не стандартизует:

- один обязательный layout коллекций для каждого проекта
- конкретную embedding-модель или размер чанка (project-local конфиг)
- автоматическое продвижение результатов поиска по коду в каноническую документацию
- развёртывание через Docker по умолчанию (Docker остаётся opt-in для shared/CI)

## AI knowledge stack

`chroma` — один слой четырёхслойного стека с тремя раздельными векторными хранилищами:

| Слой | Инструмент | Назначение | Хранилище |
|---|---|---|---|
| 0 | `docs/` (Git) | durable source of truth | Markdown |
| 1 | Basic Memory | retrieval по документации | sqlite + embeddings BM |
| 2 | ConPort | transient operational context | sqlite ConPort (+ внутр. векторы) |
| 3 | Chroma | семантический поиск по коду | sqlite Chroma PersistentClient |

Внутренние векторы ConPort, индекс кода Chroma и embeddings Basic Memory никогда не смешиваются. Раздельность удерживает каждый retrieval-слой точным, а не шумным.

## Freshness-gated запросы

Запросы Chroma должны идти через обёртку, которая обновляет индекс перед запросом и блокируется при ошибке обновления. Это предотвращает retrieval против устаревшего индекса после изменений кода.

Эталонная обёртка предоставляется как managed-шаблон в `.ai-standards/scripts/code_index.py`, синхронизируемый `ai-sync sync-templates` при включённой `chroma`.

## Similarity — не полнота

Similarity-поиск сужает исследование, но не доказывает полноту. Исчерпывающие или критичные к корректности утверждения требуют exact search (`grep`/`rg`) плюс build, type или static-проверки. Относитесь к semantic-хитам как к кандидатам для подтверждения, а не как к авторитетным ответам.

## Модель индексации

- Инкрементальная по content-hash: неизменённые файлы пропускаются.
- Atomic resumable-манифест: прерванная сборка возобновляется из манифеста без пере-embedding готовых файлов.
- Cross-file batched upsert: embeddings отправляются через границы файлов батчами, а не по одному upsert на файл, чтобы избежать бутылочного горлышка стоимости запуска per-upsert.

## Режим развёртывания

Предпочитайте локальный `PersistentClient` (embedded-хранилище, без серверного процесса, без порта) для рабочих пространств разработки. Docker используйте только для shared/CI-сценариев.

Управляемая ai-standards инфраструктура живёт под `.ai-standards/`:

```
.ai-standards/
├── scripts/code_index.py   # managed template (обобщённая эталонная обёртка)
├── code-index.toml         # managed starter-config (collections, roots, chunking)
├── chroma/                 # runtime sqlite (gitignored)
└── state/                  # manifest.json (gitignored)
```

## Навык развёртывания

При включённой `chroma` `ai-sync sync-templates` также распространяет навык развёртывания (`deploy-ai-knowledge-stack`) на агентов, перечисленных в манифесте. Навык разворачивает весь стек (ConPort, Basic Memory, Chroma) в порядке, устраняющем типичные гонки настройки. Подробности — в decision record.

## Связь с другими фичами

- `conport` остаётся transient operational memory и handoff-хранилищем.
- `basic-memory` остаётся retrieval-слоем над Git-tracked Markdown-документацией.
- `structured-artifacts` определяет, какие Markdown-артефакты считаются каноническими.
- `design-first-collaboration` удерживает интент и границы явными до развёртывания.

`chroma` дополняет эти фичи, делая исходный код семантически доступным для поиска. Он не заменяет exact search, build-верификацию или каноническую документацию.

## Пример манифеста

```toml
features = [
  "conport",
  "basic-memory",
  "chroma",
  "structured-artifacts",
]
```

## Практические подсказки для промптов

Хорошие промпты:

- `Обнови индекс кода, затем найди модули, обрабатывающие отмену заказа.`
- `Используй семантический поиск для сужения, затем подтверди exact grep и сборкой.`
- `Относись к этим similarity-хитам как к кандидатам, а не к полному списку.`

Избегать:

- `Доверяй результату семантического поиска как исчерпывающему.`
- `Запрашивать Chroma без обновления после большого изменения.`
- `Смешивать индекс кода с хранилищами ConPort или Basic Memory.`

Предпочитать:

- `Обновить, запросить, затем верифицировать кандидатов exact search и build-проверками.`
