# Structured Artifacts Usage Guide

Англоязычный оригинал: [structured-artifacts-usage.md](structured-artifacts-usage.md)

Этот документ объясняет, как использовать feature `structured-artifacts` из `ai-standards` в подключаемых проектах.

`structured-artifacts` стандартизует небольшой набор Markdown-артефактов, которые улучшают планирование, границы модулей и историю решений без превращения репозитория в XML-driven workflow.

## Цели

Используйте `structured-artifacts`, когда хотите, чтобы агент или команда:

- планировали нетривиальные изменения до реализации
- явно фиксировали границы модуля и инварианты
- держали устойчивые проектные решения видимыми в Git
- при необходимости добавляли карту для действительно трудно читаемых потоков

Типичные результаты:

- меньше скрытых изменений объёма задачи
- лучше review до начала кодинга
- более ясные ownership boundaries и non-goals для важных модулей
- меньше необходимости восстанавливать старую логику решений по истории коммитов

## Что входит в feature

Возможность стандартизует следующие артефакты:

- `change-plan.md` для планирования нетривиальной реализации
- `MODULE_CONTRACT.md` для крупных, рискованных, разделяемых или архитектурно неочевидных модулей
- `decision-record.md` для устойчивых архитектурных или операционных решений
- `module-map.md` как опциональное средство навигации по orchestration-heavy или integration-heavy потокам

Все четыре шаблона используют обычный Markdown и рассчитаны на то, чтобы быть короткими, review-friendly и локально адаптируемыми.

## Что feature сознательно отвергает

Не считайте следующую практику shared default:

- XML-heavy planning documents
- pseudo-XML knowledge overlays вокруг кода
- обязательные code graphs для каждой задачи
- повсеместные block markers или file-local semantic scaffolding
- process files для маленьких локальных правок с очевидной проверкой

Если какому-то проекту действительно нужны более тяжёлые артефакты, держите это как локальную политику проекта, а не переносите в `ai-standards`.

## Когда включать feature

Добавляйте `structured-artifacts`, когда проект регулярно использует агентов для:

- архитектурных и интеграционных изменений
- межмодульных рефакторингов
- планирования миграций
- review-sensitive изменений, где инварианты нужно проговаривать заранее
- кодовых баз, где легко ошибиться с границами модулей

Эта возможность менее полезна для:

- крошечных локальных исправлений
- очевидных механических правок
- изменений, чей эффект и проверка спокойно помещаются в обсуждение задачи без дополнительных файлов

## Пример manifest

```toml
features = [
  "conport",
  "design-first-collaboration",
  "reasoning-hygiene",
  "structured-artifacts",
]
```

## Когда создавать каждый артефакт

Создавайте change plan, когда:

- задача меняет поведение в нескольких слоях
- несколько зависимых шагов должны выполняться в безопасном порядке
- есть миграционный, rollout или compatibility risk
- review до реализации заметно снижает риск

Создавайте module contract, когда:

- у модуля есть важные инварианты
- его будут менять несколько разработчиков или агентов
- границу ownership легко размыть
- модуль достаточно велик, и non-goals надо фиксировать явно

Создавайте decision record, когда:

- команда выбирает между значимыми альтернативами
- ответ на вопрос "почему мы сделали именно так?" будет важен позже
- решение должно пережить code review и просмотр истории репозитория

Создавайте module map только когда:

- поток orchestration-heavy
- модуль затрагивает много зависимостей или side effects
- онбординг или review регулярно стопорятся на понимании одной и той же области

## Как понимать модуль

Считайте модулем минимальную единицу изменения, для которой можно внятно сформулировать один контракт ответственности и один набор инвариантов.

Хорошие границы модуля обычно имеют:

- одну основную причину для изменения
- именованные inputs, outputs и side effects
- ограниченную поверхность зависимостей
- проверку, которую можно обсуждать без удержания в голове всей системы

Плохие границы модуля обычно выглядят так:

- "эта область делает бизнес-логику"
- контракт описывает несколько несвязанных ответственностей
- инварианты расползаются на полкодовой базы
- любое изменение требует заново объяснять ещё три соседние области

## Связь с другими feature

- `design-first-collaboration` задаёт, когда intent, boundaries и non-goals должны быть вынесены наружу.
- `reasoning-hygiene` улучшает качество анализа, стоящего за этими артефактами.
- `conport` хранит active context, progress и развивающуюся project memory.
- `structured-artifacts` добавляет Git-reviewable документы для планов, контрактов и устойчивых решений.

## ConPort и decision records

Используйте ConPort для:

- active context
- недавнего прогресса
- извлечённых уроков
- рабочей rationale, которая ещё может меняться

Используйте decision record для:

- устойчивого repository-facing решения
- выбора, на который должны ссылаться будущие review или refactoring tasks
- оформленного анализа альтернатив, который должен жить вместе с кодовой базой

Не нужно зеркалить каждую запись ConPort в decision record.

## Практика внедрения

Начинайте с малого:

1. Сначала используйте `change-plan.md`.
2. Добавляйте `MODULE_CONTRACT.md` только для одного-двух действительно рискованных модулей.
3. Вводите decision records только там, где регулярно теряется ответ на вопрос "почему".
4. Добавляйте `module-map.md` только после появления реальной навигационной проблемы.

Цель не в том, чтобы нарастить бумажную нагрузку. Цель в том, чтобы уменьшить неоднозначность ровно там, где она дорого обходится.
